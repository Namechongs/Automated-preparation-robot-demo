# 涂层自动配制系统说明文档

## 1. 项目概述

本项目是一个面向涂层配制场景的桌面控制系统，使用 `PyQt5` 构建图形界面，调用 DeepSeek 大模型生成结构化配方，并将生成结果转换为机械臂、蠕动泵和搅拌器可执行的动作序列。

系统目标不是单纯“生成配方”，而是把“需求理解 -> 配方生成 -> 安全校验 -> 自动纠错 -> 物理执行 -> 日志归档”串成一条闭环流程。

当前实现具备以下特性：

- 基于自然语言需求生成 JSON 配方。
- 在执行前对配方进行多层安全校验。
- 对不合格配方进行自动纠错，最多重试 3 次。
- 使用独立线程处理模型调用和动作执行。
- 支持仿真模式与真实硬件模式。
- 支持串口扫描、日志落盘、配方归档和界面状态灯显示。

## 2. 系统架构

系统可以拆分为四层：

### 2.1 交互层

由 `main.py` 和 `main_ui.py` 组成，负责：

- 接收用户输入
- 展示生成的 JSON 配方
- 展示运行日志和执行状态
- 触发模型调用、保存配方和执行流程
- 切换模型与串口

### 2.2 配方生成层

由 `main.py` 中的系统提示词和 `ApiWorker` 构成，负责：

- 维护对话历史
- 调用 DeepSeek 接口生成结构化 JSON
- 在模型输出不合法时发起自动纠错

### 2.3 配方校验层

由 `vaildator.py` 组成，负责：

- 校验顶层 JSON 结构
- 校验物料库和泵号映射
- 校验比例区间、加料顺序、必须最后加料约束
- 校验路径安全、搅拌位置、夹爪状态和物料守恒

### 2.4 硬件执行层

由 `robot.py` 和 `dobot_api.py` 组成，负责：

- 控制 Dobot 机械臂运动
- 控制夹爪开合
- 通过 Modbus 控制蠕动泵
- 通过串口协议控制搅拌器
- 在程序退出时执行硬件关闭

## 3. 核心文件说明

## 3.1 [main.py](./main.py)

主程序入口，承担界面逻辑和业务流程编排。

### 主要职责

- 初始化日志系统
- 初始化主窗口和 LED 状态显示
- 刷新串口列表
- 创建 `RobotController`
- 构建 DeepSeek 系统提示词
- 管理多轮对话历史
- 启动模型线程和执行线程
- 执行自动纠错和拦截保护
- 归档配方 JSON

### 主要类

#### `LogSignaller`

Qt 信号转发类，用于把日志消息安全地从工作线程发送到 GUI。

#### `QtLogHandler`

继承自 `logging.Handler`，将日志同步显示到界面右下角文本框，并按级别着色：

- `INFO`：绿色
- `WARNING`：橙色
- `ERROR / CRITICAL`：红色

#### `ApiWorker(QThread)`

负责异步调用 DeepSeek 接口，避免主线程卡顿。

输入：

- OpenAI 兼容客户端
- 当前模型名
- 对话历史

输出信号：

- `finished(str)`：返回模型文本
- `error(str)`：返回异常信息

#### `ExecuteWorker(QThread)`

负责异步执行已通过校验的方案列表。

执行过程中会：

- 按步骤调用 `RobotController.execute_step`
- 推送计划状态更新
- 更新系统、机械臂、泵和搅拌器的 LED 状态

输出信号：

- `finished(str)`
- `error(str)`
- `step_update(str)`
- `led_update(str, str)`

#### `MainWindow(QMainWindow)`

主窗口控制器，负责连接所有 UI 行为。

### 关键流程

#### 配方生成

1. 读取用户输入。
2. 将用户消息追加到 `chat_history`。
3. 创建 `ApiWorker` 调用 DeepSeek。
4. 清洗返回结果中的 Markdown 代码块标记。
5. 尝试解析 JSON 并补充 `created_at`。
6. 调用 `validator_formula` 执行校验。
7. 若失败则进入自动纠错。
8. 若成功则格式化后显示在界面。

#### 自动纠错

当模型结果未通过校验时：

1. `auto_correct_count` 加 1。
2. 记录错误详情。
3. 将错误列表拼接成新的纠错提示。
4. 把错误结果和纠错提示继续喂给模型。
5. 重新启动 `ApiWorker`。

若连续失败达到 3 次，则：

- 触发最高级别拦截提示
- 停止本次自动纠错
- 防止不安全配方进入执行阶段

#### 执行配方

1. 从左侧文本框读取 JSON。
2. 再次调用 `validator_formula`。
3. 若失败，直接拦截并弹框。
4. 若成功，启动 `ExecuteWorker`。
5. 执行完成后恢复按钮状态并提示结果。

#### 保存配方

点击“保存配方”后：

- 读取当前 JSON 文本
- 提取 `task_name`
- 过滤非法文件名字符
- 以 `{task_name}_{timestamp}.json` 形式写入 `保存的配方/`

## 3.2 [robot.py](./robot.py)

硬件控制核心模块。

### `StirrerController`

通过串口与搅拌器通信，支持：

- 握手检测
- 设置转速
- 启停搅拌
- 查询实际转速
- 等待转速到位
- 按设定转速与时长执行完整搅拌流程

### `RobotController`

统一封装机械臂、泵和搅拌器控制逻辑。

### 当前关键配置

- 机械臂默认 IP：`192.168.201.1`
- Dashboard 端口：`29999`
- Feedback 端口：`30003`
- 蠕动泵：Modbus 串口通信
- 搅拌器：串口通信

### 点位定义

系统当前内置以下点位：

- `safe`
- `arm_start`
- `arm_startforward`
- `arm_startup`
- `arm_stirup`
- `arm_stir`
- `arm_endup`
- `arm_end`
- `pump_1`
- `pump_2`
- `pump_3`
- `pump_4`

### 主要方法

#### `start(port_pump, port_stir)`

根据当前模式初始化硬件连接。

仿真模式下：

- 不真正发出硬件动作
- 仅初始化状态并输出日志

真实模式下：

- 请求机械臂控制权
- 清错、使能
- 配置末端 485 通信
- 创建 Modbus 通道
- 初始化夹爪参数
- 建立泵与搅拌器通信

#### `move_to(point_name, move_model)`

控制机械臂移动到指定点位。

- `J`：关节运动
- `L`：直线运动

当前代码中：

- `arm_startforward`
- `arm_stir`
- `arm_end`

默认使用 `L` 运动，其余点位使用 `J` 运动。

#### `grip(state)`

控制夹爪开合。

- `open`
- `close`

#### `pump(pump_id, amount_ml)`

按毫升数控制指定泵出料。

主要流程：

1. 设置泵工作模式
2. 配置管径和方向
3. 将目标体积换算为寄存器参数
4. 启动出料
5. 按估算时长阻塞等待
6. 停止并执行回吸

#### `shutdown()`

退出程序时停止搅拌器、关闭泵、关闭串口，并尝试让机械臂下使能。

#### `execute_step(step)`

根据 JSON 中的 `action` 字段分发到对应控制函数：

- `move`
- `grip`
- `pump`
- `stir`

## 3.3 [vaildator.py](./vaildator.py)

配方安全校验模块，是系统最重要的保护层之一。

对外暴露：

```python
validator_formula(data) -> (bool, list)
```

返回：

- `True, []`：校验通过
- `False, error_list`：校验失败及原因

### 当前物料库配置

| 泵号 | 物料 | ratio_min | ratio_max | must_be_last | need_stir_after |
| --- | --- | --- | --- | --- | --- |
| 1 | 丙烯酸乳液 | 0.40 | 0.70 | True | False |
| 2 | 氟化丙烯酸消光剂+消泡剂 | 0.01 | 0.10 | False | True |
| 3 | 炭黑分散液 | 0.00 | 0.50 | False | False |
| 4 | 去离子水 | 0.00 | 0.30 | False | False |

总量上限：`100 ml`

### 校验维度

#### 顶层结构校验

必须存在以下字段：

- `task_name`
- `requirement`
- `formula_reasoning`
- `plans`

#### 方案结构校验

每个 `plan` 必须包含：

- `plan_id`
- `plan_name`
- `plan_reasoning`
- `materials`
- `steps`

#### 物料层校验

检查内容包括：

- 泵号是否合法
- 物料名称是否在物料库中
- 物料是否与固定泵位匹配
- `total_amount_ml` 是否大于 0
- 总量是否超过 100 ml
- 是否提供 `add_order`

#### 高级规则校验

通过以下函数实现：

- `check_ratio`
- `check_add_order`
- `check_must_be_last`

对应检查：

- 各物料比例是否落在允许区间
- 实际 `pump` 顺序是否符合 `add_order`
- `must_be_last=True` 的物料是否最后出料

#### 动作序列校验

主要检查：

- 夹爪是否重复关闭或重复打开
- 抬起烧杯前是否先到 `arm_startforward`
- 下降到 `arm_stir` 前是否先经过 `arm_stirup`
- 下降到 `arm_end` 前是否先经过 `arm_endup`
- 从 `arm_stir` 离开时是否先回 `arm_stirup`
- 从 `arm_end` 离开时是否先回 `arm_endup`
- `pump` 动作时当前位置是否为对应 `pump_x`
- 出料时机械臂是否持有容器
- 搅拌动作是否发生在 `arm_stir`
- 若某物料要求搅拌，后续下一个 `pump` 前是否出现 `stir`
- `steps` 中累计出料量是否与 `materials` 声明一致
- 执行完成后是否回到 `safe`
- 执行完成后是否释放容器

## 4. 模型集成说明

项目通过 OpenAI 兼容接口调用 DeepSeek。

当前配置位于 `main.py`：

```python
API_KEY = os.getenv("DEEPSEEK_API_KEY")
client = OpenAI(
    api_key=API_KEY,
    base_url="https://api.deepseek.com"
)
```

### 当前可切换模型

- `deepseek-reasoner`
- `deepseek-chat`

默认启动模型为：

```python
deepseek-reasoner
```

## 5. 运行流程

## 5.1 启动阶段

程序启动后会：

1. 初始化日志系统
2. 创建运行日志目录和配方目录
3. 扫描本机串口
4. 初始化 LED 为待机状态
5. 创建 `RobotController`
6. 初始化系统对话上下文

## 5.2 串口切换

当用户切换泵或搅拌器串口时：

- 若当前正在执行任务，则不允许切换
- 若存在可用串口，则重新调用 `robot.start(port_pump, port_stir)`

## 5.3 配方生成阶段

1. 用户输入需求并点击“解析”。
2. `ApiWorker` 请求模型。
3. 返回 JSON 后进入安全校验。
4. 若失败则自动纠错。
5. 若成功则在界面中格式化展示。

## 5.4 执行阶段

1. 用户点击“开始执行”。
2. 系统重新解析左侧 JSON。
3. 再次执行 `validator_formula`。
4. 通过后由 `ExecuteWorker` 逐步执行。
5. 执行期间同步更新状态灯和日志。

## 5.5 关闭阶段

窗口关闭时：

- 停止模型线程
- 停止执行线程
- 调用 `_shutdown_hardware()`
- 由 `robot.shutdown()` 做硬件收尾

## 6. 数据格式

系统要求模型输出严格 JSON，核心结构如下：

```json
{
  "task_name": "任务名称",
  "requirement": "用户需求原文",
  "formula_reasoning": "配方推理说明",
  "plans": [
    {
      "plan_id": 1,
      "plan_name": "方案A",
      "plan_reasoning": "该方案说明",
      "materials": [
        {
          "pump_id": 1,
          "material": "丙烯酸乳液",
          "total_amount_ml": 60,
          "add_order": 3
        }
      ],
      "steps": [
        {
          "step_id": 1,
          "action": "move",
          "target": "safe"
        },
        {
          "step_id": 2,
          "action": "grip",
          "state": "close"
        },
        {
          "step_id": 3,
          "action": "pump",
          "pump_id": 1,
          "amount_ml": 60
        },
        {
          "step_id": 4,
          "action": "stir",
          "duration_seconds": 120,
          "speed_rpm": 600
        }
      ]
    }
  ]
}
```

## 8. 仿真模式说明

当前主程序默认使用：

```python
sim = True
```

仿真模式下：

- 机械臂移动以日志和 `sleep` 代替
- 泵出料以时间模拟代替
- 搅拌器不进行真实串口控制
- 适合测试界面流程、模型输出和校验器逻辑

若要切换到真实硬件模式，需要：

1. 将 `sim` 改为 `False`
2. 确认机械臂网络联通
3. 确认泵与搅拌器串口可用
4. 确认 DeepSeek API 可正常访问

## 9. 已知现状与维护建议

- 校验模块文件名为 `vaildator.py`，虽然拼写不标准，但当前主程序已依赖该文件名，不建议随意改名。
- 界面中保留了“停止”“复位”按钮，但主逻辑暂未实现。
- 主流程依赖模型输出完全遵循 JSON 结构，因此系统提示词和校验器是联动关系，修改任一方时都应同步验证。
- 若后续要扩充物料库，需要同时更新：
  - `vaildator.py` 中的 `MATERIAL_CONFIG`
  - `main.py` 中生成给模型的系统提示词
  - 物理泵位与物料映射
- 若后续要扩展点位或动作，需要同步更新：
  - `robot.py` 中的 `POINT`
  - `RobotController.execute_step`
  - `vaildator.py` 的目标点与动作校验规则
